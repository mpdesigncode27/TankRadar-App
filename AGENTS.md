## Learned User Preferences

- Git ticket branches must branch off **updated `main`** only (never off another feature branch). Prefer branch names **`feature/TAN-XX-short-slug`** — avoid Linear/GitHub-style prefixes such as `mpdesigncode27/` before the ticket slug.
- Before checking out a **new** branch while work still exists on the **current** branch, **ask the user** whether that existing branch should be **merged (or otherwise finished)** first — do not silently switch away without that check.
- Before implementing tickets that involve Tankerkönig (API shapes, networking, station/price models), read `.cursor/skills/tankerkoenig-api/SKILL.md`, verify the ticket matches the real API, and update the Linear issue if specs drift.
- When marking a Linear ticket complete, attach at least one screenshot or short screen recording of the implementation to the issue (Simulator or device), unless the work is purely non-visual—in that case attach an alternative proof such as an Xcode test-run screenshot and note it in the closing comment. Keep captures **out of git**; optional naming **`TAN-XX-short-description.png`** (or `.mov`/`.mp4`). See `.cursor/rules/ticket-branch-workflow.mdc` § „Visueller Nachweis — konkret“.
- When planning SDD-style work, track scope with Linear issues per task and keep those issues updated as implementation progresses.
- Linear issues should include explicit acceptance criteria so completion can be verified objectively.
- Linear issues should include a concise opening statement of the ticket’s goal and intended outcome.
- Expectations include strong Swift and SwiftUI quality, automated tests where appropriate, alignment with current Apple Human Interface Guidelines and Liquid Glass styling, and **WCAG 2.2 AAA color contrast** (Text ≥ 7:1 / large Text ≥ 4.5:1, UI/Graphics ≥ 3:1) — siehe `docs/A11Y.md`.
- When ticket work is finished, **squash-merge per Repo-Standard** (`gh pr merge <num> --squash --delete-branch`), das zugehörige Linear-Issue auf **Done** setzen (Abschlusskommentar mit PR-Link / Merge-Commit-SHA, Akzeptanz-Checkboxen aktualisieren) und alle vollständig gemergten lokalen/Remote-Feature-Branches prunen.
- **FuelNow UI conventions:** Tankstellen-Öffnungsstatus als **farbiger Punkt** (grün geöffnet / rot geschlossen) statt Icon; Preise auf **2 Nachkommastellen** anzeigen (z. B. `1,89`, nicht `1,859`) und **ohne** „€/l“-Label; „In Apple Maps öffnen" startet **Turn-by-turn-Navigation** zur Tankstelle (nicht nur Anzeige); Sheets nutzen ein **Close-Icon** statt eines „Fertig"-Buttons; **kein** Suchradius-Setting (Karte/Liste nutzen fest die Tankerkönig-API-Obergrenze **25 km**); **keine** Adresse im Standort-Details-Sheet.
- **FuelNow / iOS:** Nach relevanten Codeänderungen oder zur Verifikation **immer neu bauen und die App im Simulator starten** — bevorzugt `./scripts/build-and-run-simulator.sh` (legt Derived Data unter `.derived-data-ios/`, installiert auf den gewählten Simulator und startet `com.vibecoding.fuelnow`). Bei Bedarf vorher `open -a Simulator` bzw. Zielgerät per `SIMULATOR_NAME` anpassen.
- **AXe-Skill & Workflow:** Vor jedem Einsatz von AXe das Skill **`~/.agents/skills/axe/SKILL.md` vollständig lesen** (Simulator-`--udid`, Batch vs. Einzelkommandos, `describe-ui`/`screenshot` zur Verifikation, Timing/`--wait-timeout`). Install: `brew tap cameroncooke/axe && brew install axe` ([axe-cli.com](https://www.axe-cli.com/)).
  - **Wann der Agent das Skill nutzt:** Auf ausdrückliche oder implizite Aufgaben wie „AXe“, Simulator automatisieren, Tippen/Wischen/Tippen simulieren, UI-Baum auslesen, Screenshot/Video vom Simulator, Smoke-/UI-Flow skripten; bei **UI-lastigen Tickets** zur Verifikation nach Build; beim Erzeugen von **Linear-Nachweis-Screenshots** im Simulator (reproduzierbar).
  - **Projekt-Skripte (Standard):** `./scripts/build-run-and-axe.sh` (Build + Start + Batch aus `scripts/axe/fuelnow-smoke.steps`) oder `./scripts/run-axe-batch.sh` wenn die App schon läuft. Batch-Dateien: nur Interaktion + `sleep` — **kein** `screenshot` in der `.steps`-Datei; `run-axe-batch.sh` macht danach **immer** einen Abschluss-Screenshot (auch bei Batch-Fehler), Standard `scripts/axe/output/fuelnow-launch.png`.
  - **Linear-Nachweis (komfortabel):** `./scripts/capture-linear-evidence.sh TAN-XX-kurz-slug` schreibt nach `.linear-evidence/<slug>.png` und nutzt den Standard-Smoke (Einstellungen kurz öffnen/schließen → Abschlussfoto zeigt die Karte). Für einen Screenshot **mit geöffnetem Einstellungen-Sheet:** `./scripts/capture-linear-evidence.sh TAN-XX-kurz-slug --settings-sheet` (`scripts/axe/fuelnow-settings-sheet.steps`). Alternativ weiterhin manuell `AXE_SCREENSHOT_PATH` setzen.
  - **SDD Research:** `./scripts/scan-patterns.sh` — grobe Stack-Erkennung (Xcode/Swift, Tests, AXe), wenn Phase 1 eines Research-Laufs ein Scan-Skript erwartet.
  - **Linear-Nachweis (manuell):** Screenshot-Pfad setzen, z. B. `AXE_SCREENSHOT_PATH=".linear-evidence/TAN-XX-kurz-slug.png"` (Ordner `.linear-evidence/` ist gitignored). Batch-Schritte bei neuen UI-Flows in `scripts/axe/*.steps` ergänzen, wenn der Smoke-Pfad relevante Screens nicht mehr abdeckt.
  - **Wenn AXe fehlt oder nicht passt:** Hinweis aus Script/Fehlermeldung; Fallback manueller Simulator-Screenshot oder Xcode-Test-Report für nicht-visuelle Arbeit (wie in `.cursor/rules/ticket-branch-workflow.mdc`).

## Learned Workspace Facts

- FuelNow is a native iOS app that shows nearby fuel stations on a map with opening status and prices; users choose a fuel grade (e.g. Super, Super95, Diesel) in settings.
- Planned capabilities include Siri-driven queries for nearest or cheapest nearby stations, a CarPlay map-centric experience, and a subscription gate where CarPlay is enabled only for subscribers (initial placeholder pricing discussed at €6 per year).
- Planning and architecture references German station data via Tankerkönig and backend/services direction involving Appwrite (Swift SDK).
- Primary Git remote for the project is https://github.com/mpdesigncode27/TankRadar-App.git.
- App was **renamed TankRadar → FuelNow**: das Produkt/Target heißt jetzt `FuelNow`, der **Repo-Pfad** (`/Users/.../Documents/Dev/iOS/TankRadar`) und das **Git-Remote** (`TankRadar-App.git`) tragen historisch weiterhin den `TankRadar`-Namen — nicht „aufräumen“, sondern als bewussten Mismatch behandeln.
- Xcode **`PRODUCT_BUNDLE_IDENTIFIER`** for the app is **`com.vibecoding.fuelnow`** (tests use matching `…Tests` / `…UITests` IDs; ASC-konform kleingeschrieben).
- For Simulator **DEBUG** runs, a Tankerkönig key can live in **`~/.fuelnow/tankerkoenig-api-key`** on the Mac (read via **`SIMULATOR_HOST_HOME`**); details are in **`README.md`** and **`FuelNow/Support/APIKeys.example.swift`** — never commit the key file.
- **Linear-Backlog-Status (FuelNow App):** Vor jedem neuen Ticket den aktuellen Linear-Status prüfen — frühere Reihenfolge ist überholt; abgeschlossen sind u. a. der Phase-1-Settings-/Liquid-Glass-Strang (`TAN-71`/`TAN-74`/`TAN-75`/`TAN-76`/`TAN-77`/`TAN-78`), Search-Radius-Removal (`TAN-79`), A11y-WCAG-AAA (`TAN-80`), die StoreKit-Plus-Vorarbeiten (`TAN-44`/`TAN-45`/`TAN-62`) sowie CarPlay-Capability (`TAN-54`); aktueller Schwerpunkt ist **`TAN-81`** (Plus 3-Tage Free-Trial Introductory Offer) und weitere Plus-/CarPlay-Tickets.
- **FuelNow Plus / Subscription stack:** **StoreKit 2 + App Store Connect** — **kein RevenueCat**. Apples **Mindest-Free-Trial in ASC ist 3 Tage** (48 h ist nicht buchbar — Variante A festgelegt). Lokale DEBUG-Konfiguration in **`FuelNowPlus.storekit`** (im Run-Scheme referenziert). Unit-Tests nutzen `SKTestSession` mit `FuelNowPlus.storekit` als Test-Resource und sind via `.disabled(if:)` auf die OS-Version konditioniert wegen des **iOS-26.3/26.4-Bugs**, bei dem `SKTestSession` die `.storekit`-Konfig in Unit-Tests ignoriert (fix ab **iOS 26.5**).
- **Zentrale Produktspezifikation:** **`docs/PRODUCT_SPEC.md`** ist der Spec-Index (Produktüberblick, Naming/Repo-Mismatch, technische Konstanten, Kernflows, UI-Konventionen, Spec-Quellen-Tabelle) und ist aus **`README.md`** verlinkt — bei spec-relevanten Änderungen mitpflegen.
- **Linear-Evidence-Größe:** Anhänge für `create_attachment` (Linear MCP) klein halten — z. B. `sips -Z 800 -s format jpeg -s formatOptions 65 in.png --out out-mini.jpg` und davon Base64 übergeben (`base64 -i out-mini.jpg`); große PNGs/Base64-Strings stoßen sonst an die Größenlimits der MCP-/Read-Tools.
